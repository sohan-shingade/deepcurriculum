# RAG Failure Modes and Evaluation

In the last lecture, we built RAG from first principles — embeddings, vector search, retrieval, generation. It is elegant on the whiteboard. In production, it breaks in ways that are subtle, systematic, and infuriating. This lecture is about **how RAG fails** and, more importantly, **how to measure whether your RAG system is working**.

Here is the core thesis of this lecture: **you cannot improve what you cannot measure**. Every production AI system needs an evaluation framework *before* it needs a better model. This is the "evals-first" philosophy, and it is the single most important mindset difference between a hobbyist and a production AI engineer.

## The Five Failure Modes of RAG

Every RAG failure falls into one of five categories. Understanding these categories is essential for debugging — when your RAG system produces a wrong answer, your first job is to classify *which* failure mode you are observing, because the fix is completely different for each.

### Failure Mode 1: Retrieval Failure

**The right information exists in your corpus, but the retriever does not return it.**

This is the most common failure mode and usually the easiest to diagnose. Causes include:

- **Vocabulary mismatch**: The user asks "how to terminate an employee" but the document says "involuntary separation procedures." Dense embeddings help here (they capture semantic similarity), but they are not perfect — especially for domain-specific jargon.
- **Embedding model blind spots**: The embedding model was not trained on your domain. Medical, legal, and financial text has specialized semantics that general-purpose models miss.
- **Chunk boundary problems**: The relevant information spans two chunks and neither chunk alone is sufficient. The sentence "The dosage should be reduced for patients with renal impairment" might be split between a chunk about dosage and a chunk about contraindications.
- **Query-document asymmetry**: The query is a short question ("What is the TPS report deadline?") but the relevant document is a long procedural description. Short queries and long documents live in different regions of embedding space.

**Diagnosis**: Log the retrieved chunks for failed queries. If the right chunk exists but was not retrieved, you have a retrieval failure. The fix might be query expansion, hybrid search (BM25 + dense), or a domain-specific embedding model.

### Failure Mode 2: Context Window Overflow

**The retriever returns too much context, and the LLM cannot process it effectively.**

Modern LLMs have context windows of 128K or even 1M tokens, but **more context is not always better**. Liu et al. (2024) demonstrated the "lost in the middle" phenomenon: LLMs pay the most attention to the beginning and end of the context window, with significant degradation for information in the middle. If you stuff 50 chunks into the prompt, the LLM may ignore chunks 10-40 entirely.

**Diagnosis**: Test with varying numbers of retrieved chunks ($k = 3, 5, 10, 20$). If quality degrades as $k$ increases, you have context overflow. The fix is either fewer chunks (retrieve less, rank better) or context compression (extract relevant sentences from chunks before prompting).

### Failure Mode 3: Wrong Context Selection

**The retriever returns documents, but they answer a different question than the one asked.**

This is more insidious than retrieval failure because the system *looks* like it is working — it retrieves documents, generates a fluent answer — but the answer is wrong because the context was wrong.

Example: A user asks "What is our return policy for electronics?" The retriever returns the general return policy (30 days) instead of the electronics-specific policy (15 days, restocking fee). The embedding for "return policy" is similar regardless of the product category.

**Diagnosis**: Human review of retrieved chunks for a sample of queries. Automated detection via "context relevancy" metrics (see RAGAS below). The fix is often better chunking (ensure the electronics policy is a distinct chunk with clear metadata) or metadata filtering (filter by product category before vector search).

### Failure Mode 4: Hallucination Despite Context

**The retriever returns the right context, but the LLM generates an answer that contradicts or goes beyond the provided evidence.**

This is the most dangerous failure mode because it is invisible without careful checking. The LLM has the correct information in its context window and *still* produces an incorrect answer, either by:

- **Over-relying on parametric knowledge**: The model's training data contains different information than the retrieved context, and it defaults to training data.
- **Misinterpreting the context**: The model reads a table of pricing tiers and confuses which tier applies to which customer segment.
- **Confabulating details**: The context says "revenue increased" but the model fabricates a specific percentage.

**Diagnosis**: Compare the generated answer against the retrieved context using an LLM-as-judge (a second LLM call that checks faithfulness). The fix involves prompt engineering ("Answer ONLY based on the provided context"), stronger instruction-following models, or citation enforcement (require the model to cite specific sentences from the context).

### Failure Mode 5: Stale or Inconsistent Embeddings

**The vector index is out of date or inconsistently updated.**

Your company updates a pricing page. The new page is added to the corpus, but the old page is still indexed. The retriever might return either version, or both. Worse: if you re-embed the new page with a different embedding model version, it lives in a subtly different vector space than the old embeddings.

**Diagnosis**: Track document versions and embedding timestamps. The fix is an operational pipeline: on document update, delete old embeddings, re-chunk, re-embed, and insert new ones atomically. This is unglamorous plumbing, but it is essential.

## Retrieval Metrics: Measuring Search Quality

Before we can evaluate the full RAG pipeline, we need to evaluate the retriever in isolation. Given a query $q$ and a set of truly relevant documents $\mathcal{R}_q \subset \mathcal{D}$, how well does the retriever rank relevant documents?

### Precision@k

Of the top-$k$ retrieved documents, what fraction is relevant?

$$\text{Precision@}k = \frac{|\{d \in \text{top-}k : d \in \mathcal{R}_q\}|}{k}$$

If we retrieve 10 documents and 3 are relevant, $\text{Precision@}10 = 0.3$. Precision@k penalizes retrievers that return many irrelevant results. However, it does not care about the *order* of relevant results within the top-$k$.

### Recall@k

Of all truly relevant documents, what fraction appears in the top-$k$?

$$\text{Recall@}k = \frac{|\{d \in \text{top-}k : d \in \mathcal{R}_q\}|}{|\mathcal{R}_q|}$$

If there are 5 relevant documents in the corpus and 3 appear in the top-10, $\text{Recall@}10 = 0.6$. Recall is critical for RAG because missing a relevant document means the LLM cannot use that information — it is a hard failure. For most RAG systems, **optimizing recall@k matters more than precision@k**, because the LLM can often ignore irrelevant context but cannot invent missing context.

### Mean Reciprocal Rank (MRR)

MRR focuses on *where* the first relevant document appears. For a set of queries $\mathcal{Q}$:

$$\text{MRR} = \frac{1}{|\mathcal{Q}|} \sum_{q \in \mathcal{Q}} \frac{1}{\text{rank}_q}$$

where $\text{rank}_q$ is the position of the first relevant document for query $q$. If the first relevant result is at position 1, the reciprocal rank is 1.0. At position 3, it is 0.333. If no relevant result appears, the reciprocal rank is 0.

MRR is a useful metric when you typically only need *one* relevant document (e.g., factoid question answering). It heavily rewards retrievers that place at least one good result near the top.

### Normalized Discounted Cumulative Gain (NDCG)

NDCG is the gold standard for ranking evaluation because it accounts for both relevance and position. It handles graded relevance (documents can be "somewhat relevant" not just binary relevant/irrelevant).

**Discounted Cumulative Gain (DCG@k):**

$$\text{DCG@}k = \sum_{i=1}^{k} \frac{\text{rel}_i}{\log_2(i + 1)}$$

where $\text{rel}_i$ is the relevance score of the document at position $i$. The $\log_2(i+1)$ denominator discounts lower-ranked results — a relevant document at position 1 contributes much more than the same document at position 10.

**Ideal DCG (IDCG@k):** The DCG@k you would get with a perfect ranking — sort all documents by relevance and compute DCG.

**NDCG:**

$$\text{NDCG@}k = \frac{\text{DCG@}k}{\text{IDCG@}k}$$

NDCG ranges from 0 to 1, where 1 means a perfect ranking. It is the primary metric used in MTEB retrieval benchmarks and in industry search evaluation.

Let us derive a concrete example. Suppose we have 5 retrieved documents with relevance scores $[3, 0, 2, 0, 1]$ (on a 0-3 scale):

$$\text{DCG@5} = \frac{3}{\log_2 2} + \frac{0}{\log_2 3} + \frac{2}{\log_2 4} + \frac{0}{\log_2 5} + \frac{1}{\log_2 6} = 3.0 + 0 + 1.0 + 0 + 0.387 = 4.387$$

The ideal ranking would be $[3, 2, 1, 0, 0]$:

$$\text{IDCG@5} = \frac{3}{\log_2 2} + \frac{2}{\log_2 3} + \frac{1}{\log_2 4} + 0 + 0 = 3.0 + 1.262 + 0.5 = 4.762$$

$$\text{NDCG@5} = \frac{4.387}{4.762} = 0.921$$

A score of 0.921 means our ranking is close to ideal but not perfect — the relevant document with score 2 should have been ranked higher than position 3.

```python
import numpy as np
from typing import List

def precision_at_k(retrieved: List[str], relevant: set, k: int) -> float:
    """Compute Precision@k."""
    top_k = retrieved[:k]
    relevant_in_top_k = sum(1 for doc in top_k if doc in relevant)
    return relevant_in_top_k / k

def recall_at_k(retrieved: List[str], relevant: set, k: int) -> float:
    """Compute Recall@k."""
    top_k = retrieved[:k]
    relevant_in_top_k = sum(1 for doc in top_k if doc in relevant)
    return relevant_in_top_k / len(relevant) if relevant else 0.0

def mrr(queries_results: List[tuple]) -> float:
    """Compute Mean Reciprocal Rank over a set of queries.

    Args:
        queries_results: List of (retrieved_docs, relevant_set) tuples
    """
    reciprocal_ranks = []
    for retrieved, relevant in queries_results:
        rr = 0.0
        for rank, doc in enumerate(retrieved, 1):
            if doc in relevant:
                rr = 1.0 / rank
                break
        reciprocal_ranks.append(rr)
    return np.mean(reciprocal_ranks)

def ndcg_at_k(relevance_scores: List[float], k: int) -> float:
    """Compute NDCG@k given relevance scores in retrieval order."""
    scores = relevance_scores[:k]

    # DCG
    dcg = sum(score / np.log2(i + 2) for i, score in enumerate(scores))

    # IDCG — sort by relevance descending
    ideal_scores = sorted(relevance_scores, reverse=True)[:k]
    idcg = sum(score / np.log2(i + 2) for i, score in enumerate(ideal_scores))

    return dcg / idcg if idcg > 0 else 0.0

# Example
retrieved = ["doc_a", "doc_b", "doc_c", "doc_d", "doc_e"]
relevant = {"doc_a", "doc_c", "doc_f"}

print(f"Precision@3: {precision_at_k(retrieved, relevant, 3):.3f}")
print(f"Recall@3: {recall_at_k(retrieved, relevant, 3):.3f}")
print(f"Recall@5: {recall_at_k(retrieved, relevant, 5):.3f}")

queries = [
    (["doc_x", "doc_a", "doc_y"], {"doc_a"}),
    (["doc_a", "doc_b"], {"doc_a", "doc_b"}),
    (["doc_x", "doc_y", "doc_z"], {"doc_a"}),  # no relevant found
]
print(f"MRR: {mrr(queries):.3f}")

relevance = [3, 0, 2, 0, 1]
print(f"NDCG@5: {ndcg_at_k(relevance, 5):.3f}")
```

## RAGAS: A Framework for End-to-End RAG Evaluation

Individual retrieval metrics are necessary but not sufficient. We need to evaluate the full pipeline: retrieval + generation together. The RAGAS framework by Es et al. (2024) provides four complementary metrics:

### Faithfulness

Does the generated answer contain only information that is supported by the retrieved context? This detects Failure Mode 4 (hallucination despite context).

$$\text{Faithfulness} = \frac{|\text{claims in answer supported by context}|}{|\text{total claims in answer}|}$$

Implementation: An LLM extracts individual claims from the answer, then another LLM call checks each claim against the context. A faithfulness score of 1.0 means every claim is supported. This is the most critical metric — an unfaithful RAG system is worse than no RAG system, because it generates convincing falsehoods.

### Answer Relevancy

Does the generated answer actually address the question? An answer can be faithful to the context but completely miss the point of the query.

$$\text{Answer Relevancy} = \frac{1}{N} \sum_{i=1}^{N} \text{sim}(q, \hat{q}_i)$$

where $\hat{q}_i$ are questions generated from the answer. If the answer is relevant, the generated questions should be similar to the original query.

### Context Relevancy

Of the retrieved context, how much is actually relevant to the question? This measures retrieval precision from the generation perspective.

$$\text{Context Relevancy} = \frac{|\text{relevant sentences in context}|}{|\text{total sentences in context}|}$$

Low context relevancy means the retriever is flooding the LLM with irrelevant information — a precursor to Failure Mode 2 (context overflow).

### Context Recall

Does the retrieved context contain all the information needed to answer the question? This measures retrieval recall from the generation perspective.

$$\text{Context Recall} = \frac{|\text{ground truth claims supported by context}|}{|\text{total ground truth claims}|}$$

This requires a ground truth answer, which makes it more expensive to compute but essential for comprehensive evaluation.

The RAGAS framework is elegant because each metric targets a specific failure mode. Low faithfulness → Failure Mode 4. Low context relevancy → Failure Mode 3. Low context recall → Failure Mode 1. This gives you a direct diagnostic path from metrics to fixes.

## LLM-as-Judge: Scalable Evaluation

A fundamental problem with RAG evaluation is that many metrics require **human-level understanding** to compute. Is this claim supported by the context? Is this answer relevant to the question? Hiring human annotators is expensive and slow. The practical solution is to use an LLM as a judge.

### Pointwise Scoring

Ask an LLM to rate a single answer on a scale (e.g., 1-5 for relevance, accuracy, completeness). This is the simplest approach but suffers from calibration issues — different prompts and models produce systematically different score distributions.

```
Rate the following answer on a scale of 1-5 for accuracy:
Question: {question}
Context: {context}
Answer: {answer}
Score (1-5):
```

### Pairwise Comparison

Ask an LLM to compare two answers and select the better one. This is more reliable than pointwise scoring because relative comparison is easier than absolute scoring. Zheng et al. (2023) showed in the "Judging LLM-as-a-Judge" paper that pairwise comparison has higher agreement with human judges than pointwise scoring.

```
Which answer better addresses the question?
Question: {question}
Answer A: {answer_a}
Answer B: {answer_b}
Better answer (A or B):
```

**Position bias warning**: LLMs tend to prefer the first answer presented. Always evaluate both orderings (A first, then B first) and check for consistency.

### Reference-Based Evaluation

Compare the generated answer against a gold-standard reference answer. This is the most rigorous approach but requires creating reference answers — expensive upfront but invaluable for regression testing.

```
Given the reference answer, evaluate the candidate answer for:
1. Factual consistency (does it contain any facts not in the reference?)
2. Completeness (does it cover all key points from the reference?)
3. Conciseness (is it unnecessarily verbose?)

Reference: {reference_answer}
Candidate: {candidate_answer}
```

The key insight: **use reference-based evaluation for your test set and pointwise/pairwise for production monitoring**. You cannot create reference answers for every production query, but you can run an LLM judge on a sample of real traffic.

## Synthetic Evaluation Data Generation

You need evaluation data. Lots of it. And it needs to cover the distribution of real user queries. Three strategies:

### Strategy 1: Generate from Your Corpus

Take a document from your corpus, use an LLM to generate questions that can be answered by that document, then use the document as the ground truth context and generate a reference answer. This gives you (query, context, reference_answer) triples for free.

The risk: generated questions may not reflect how real users ask questions. A generated question like "What are the three components of the retrieval pipeline described in Section 3.2?" is too specific. Real users ask "how do I search my documents?"

### Strategy 2: Seed with Real Queries, Augment with LLM

Collect a small set of real user queries (even 50-100 is valuable), manually annotate the correct answers, then use an LLM to generate paraphrases and variations. This preserves the distribution of real queries while scaling your eval set.

### Strategy 3: Adversarial Generation

Deliberately generate hard cases: queries that are similar to your domain but unanswerable from the corpus (test for hallucination), queries that require combining multiple documents (test for multi-hop retrieval), and queries with common misspellings or colloquialisms (test for robustness).

```python
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class EvalExample:
    """A single RAG evaluation example."""
    query: str
    relevant_doc_ids: List[str]
    reference_answer: str
    difficulty: str  # "easy", "medium", "hard"
    category: str    # "factoid", "multi-hop", "unanswerable"

def create_eval_set_from_corpus(
    documents: List[dict],  # {"id": str, "text": str}
    queries_per_doc: int = 2
) -> List[EvalExample]:
    """Generate synthetic eval examples from a document corpus.

    In practice, each query/answer would be generated by an LLM.
    Here we simulate the structure.
    """
    eval_set = []

    for doc in documents:
        # Simulate LLM-generated questions from this document
        # In production: prompt an LLM with the doc text
        for i in range(queries_per_doc):
            example = EvalExample(
                query=f"Synthetic question {i+1} about: {doc['text'][:50]}",
                relevant_doc_ids=[doc["id"]],
                reference_answer=f"Answer derived from doc {doc['id']}",
                difficulty="medium",
                category="factoid"
            )
            eval_set.append(example)

    return eval_set

# Demo
docs = [
    {"id": "doc_001", "text": "RAG combines retrieval with generation to ground LLM outputs in evidence."},
    {"id": "doc_002", "text": "Vector databases use ANN algorithms like HNSW for fast similarity search."},
    {"id": "doc_003", "text": "Chunking strategy significantly impacts retrieval quality in RAG systems."},
]

eval_set = create_eval_set_from_corpus(docs, queries_per_doc=2)
print(f"Generated {len(eval_set)} eval examples")
for ex in eval_set[:3]:
    print(f"  Query: {ex.query}")
    print(f"  Relevant: {ex.relevant_doc_ids}")
    print(f"  Category: {ex.category}")
    print()
```

## The Evals-First Philosophy

Let me be direct about something: **the number one mistake** I see in AI engineering teams is building features before building evals. They fine-tune a prompt, eyeball a few examples, ship it, and then wonder why users complain. This is the equivalent of writing a web application without tests and deploying to production on a Friday afternoon.

The evals-first workflow:

1. **Define your metrics** before writing any RAG code. What is faithfulness? What is "good enough" retrieval? What precision@k do you need?
2. **Build your eval set** before optimizing. You need at least 50-100 representative query-answer pairs. More is better. Diverse is essential.
3. **Baseline everything**. Run your simplest possible RAG pipeline (naive RAG, default chunking, default embedding model) against your eval set. Record all metrics. This is your floor.
4. **Change one thing at a time.** Switch the embedding model. Measure. Change the chunk size. Measure. Add a reranker. Measure. If a change does not move your metrics, revert it.
5. **Monitor in production.** Sample real queries, run LLM-as-judge, track metrics over time. Alert on regressions.

This is not glamorous. It is not as exciting as implementing the latest multi-hop retrieval architecture from a blog post. But it is what separates systems that work from systems that "work on my machine."

The RAG systems that succeed in production are not the ones with the most sophisticated architectures. They are the ones with the best evaluation frameworks. Build the measurement apparatus first, and the engineering decisions become obvious.

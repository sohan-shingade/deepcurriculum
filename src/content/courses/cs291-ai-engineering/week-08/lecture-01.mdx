# Observability for AI Systems

Your AI system is in production. Users are sending requests. The model is returning completions. Revenue is flowing. And you have almost no idea what is actually happening inside the system.

This is the state of most deployed AI applications today. Teams build sophisticated prompt chains, deploy RAG pipelines with multiple retrieval stages, and wire up agent loops with tool calling — then monitor them with the same infrastructure they use for CRUD APIs. They watch HTTP status codes and average latency. When something goes wrong, they grep through logs for error messages. When the CEO asks "why did the chatbot tell a customer our product causes cancer?", the answer is a shrug and a Slack thread.

Traditional observability was designed for deterministic systems. A web server receives a request, executes a known code path, queries a database, and returns a response. The inputs map predictably to outputs. You can reproduce bugs. You can trace failures to root causes. AI systems violate every one of these assumptions. The same input can produce different outputs. The "code path" is a neural network with billions of parameters. Failures are often semantic — the system returns a 200 OK with a response that is confidently, catastrophically wrong.

This lecture is about building observability that actually works for AI systems. We will adapt the classical three pillars — traces, metrics, and logs — to handle the unique challenges of LLM-based applications. By the end, you will know how to instrument an AI system so that when something breaks (and it will), you can figure out what happened, why, and how to prevent it from happening again.

---

## 1. Why AI Systems Break Traditional Observability

Let us start with a concrete example. You have deployed a RAG-based customer support bot. A user asks: "Can I return my order after 90 days?" Your system retrieves three context documents, constructs a prompt, calls GPT-4, and returns an answer. The answer is wrong — it says returns are accepted within 120 days when the actual policy is 30 days.

In a traditional system, you would look at the HTTP response code (200), the latency (acceptable), the database query (returned results), and conclude everything is fine. The observability stack saw no anomaly. But the system failed catastrophically — it gave a user incorrect information that could create legal liability.

The failure happened across multiple layers that traditional observability does not inspect:

1. **Retrieval failure**: The vector search returned documents about a different product's return policy.
2. **Context assembly failure**: The most relevant document was truncated to fit the context window.
3. **Generation failure**: The model hallucinated a specific number (120 days) that appeared nowhere in the provided context.

To diagnose this, you need to see: what query was sent to the vector database, what documents came back and their relevance scores, how the prompt was assembled, what the full prompt looked like, what the model returned, and how the response was post-processed. This is fundamentally different from monitoring a REST API.

### 1.1 The Unique Challenges

**Non-determinism.** Even with temperature 0, LLM outputs are not perfectly reproducible across API versions, hardware, and time. The same prompt can yield different responses. This means you cannot simply "replay" a failure — you must capture the full input-output pair at the time it happened.

**Semantic failures.** The system can fail while every technical metric looks healthy. Latency is normal, tokens are within budget, no exceptions were thrown — but the output is wrong, harmful, or off-topic. You need quality metrics, not just operational metrics.

**Cascading opacity.** In an agent loop, step $n$ depends on the output of step $n-1$, which was generated by a model. A small error in step 2 can compound through steps 3, 4, and 5 into a completely derailed execution. You need trace-level visibility into every step.

**Cost as a first-class concern.** Every LLM call costs money. A poorly constructed prompt that uses 4000 tokens when 400 would suffice is not a performance bug — it is a financial one. Cost must be tracked with the same granularity as latency.

## 2. The Three Pillars, Reimagined

The classical observability pillars — traces, metrics, and logs — were defined by Cindy Sridharan in *Distributed Systems Observability* (2018). They remain the right framework, but their implementation must change substantially for AI systems.

### 2.1 Traces for AI: Following the Thought Process

In a microservices architecture, a trace follows a request across service boundaries. In an AI system, a trace follows a request through the **reasoning process**: embedding generation, retrieval, prompt construction, LLM invocation, output parsing, tool calls, and validation.

An AI trace has a hierarchical structure:

```
Trace: user_query "How do I reset my password?"
├── Span: query_embedding (12ms, model=text-embedding-3-small, tokens=8)
├── Span: vector_search (45ms, results=5, top_score=0.89)
├── Span: context_assembly (2ms, total_context_tokens=1200)
├── Span: llm_call (890ms, model=gpt-4-turbo, input_tokens=1450, output_tokens=180, cost=$0.0052)
│   ├── Span: prompt_template_render (1ms, template=support_v3)
│   └── Span: api_call (887ms, status=200, finish_reason=stop)
├── Span: output_parse (3ms, format=json, valid=true)
└── Span: guardrail_check (15ms, passed=true, checks=[pii, toxicity, relevance])
```

Each span captures the specific metadata relevant to that operation type. An LLM call span must capture the model name, input and output token counts, finish reason, temperature, and cost. A retrieval span must capture the query, the number of results, and their relevance scores.

For **agent loops**, the trace structure becomes recursive. Each agent step is a span that may itself contain sub-spans for tool calls, each of which may trigger further LLM calls:

```
Trace: agent_task "Book a flight from SFO to JFK"
├── Span: agent_step_1 (thought="I need to search for flights")
│   ├── Span: tool_call (tool=search_flights, args={from: SFO, to: JFK, date: 2026-04-15})
│   │   └── Span: api_call (external_service=amadeus, latency=340ms)
│   └── Span: llm_call (model=claude-3.5-sonnet, tokens_in=2100, tokens_out=250, cost=$0.009)
├── Span: agent_step_2 (thought="Found 3 options, need to compare prices")
│   └── Span: llm_call (model=claude-3.5-sonnet, tokens_in=2800, tokens_out=400, cost=$0.013)
└── Span: agent_step_3 (thought="Recommending the 2pm departure")
    └── Span: llm_call (model=claude-3.5-sonnet, tokens_in=3200, tokens_out=300, cost=$0.012)
Total: 3 steps, 3 LLM calls, 1 tool call, $0.034, 2.1s
```

### 2.2 Implementing Trace Decorators

The most practical approach to AI tracing is **decorator-based instrumentation**. You wrap your LLM call functions so that every invocation automatically captures metadata without modifying business logic.

```python
import time
import json
from dataclasses import dataclass, field
from typing import Any, Optional

@dataclass
class Span:
    name: str
    span_type: str
    start_time: float = 0.0
    end_time: float = 0.0
    attributes: dict = field(default_factory=dict)
    children: list = field(default_factory=list)
    parent: Optional['Span'] = None

    @property
    def duration_ms(self) -> float:
        return (self.end_time - self.start_time) * 1000

    def to_dict(self) -> dict:
        result = {
            "name": self.name,
            "type": self.span_type,
            "duration_ms": round(self.duration_ms, 2),
            "attributes": self.attributes,
        }
        if self.children:
            result["children"] = [c.to_dict() for c in self.children]
        return result


@dataclass
class Trace:
    trace_id: str
    root_span: Optional[Span] = None
    _current_span: Optional[Span] = None

    def start_span(self, name: str, span_type: str, attributes: dict = None) -> Span:
        span = Span(
            name=name,
            span_type=span_type,
            start_time=time.time(),
            attributes=attributes or {},
            parent=self._current_span,
        )
        if self.root_span is None:
            self.root_span = span
        if self._current_span is not None:
            self._current_span.children.append(span)
        self._current_span = span
        return span

    def end_span(self) -> None:
        if self._current_span is not None:
            self._current_span.end_time = time.time()
            self._current_span = self._current_span.parent

    def to_dict(self) -> dict:
        return {
            "trace_id": self.trace_id,
            "root": self.root_span.to_dict() if self.root_span else None,
        }


# Usage: tracing an LLM call
trace = Trace(trace_id="req-abc-123")

# Start a top-level span
trace.start_span("handle_query", "request", {"user_id": "u-42", "query": "What is RAG?"})

# Nested span for retrieval
trace.start_span("vector_search", "retrieval", {"index": "docs-v2", "top_k": 5})
time.sleep(0.01)  # simulate retrieval
trace._current_span.attributes["results_count"] = 5
trace._current_span.attributes["top_score"] = 0.91
trace.end_span()

# Nested span for LLM call
trace.start_span("llm_call", "llm", {
    "model": "gpt-4-turbo",
    "temperature": 0.0,
    "input_tokens": 1450,
})
time.sleep(0.05)  # simulate LLM call
trace._current_span.attributes["output_tokens"] = 180
trace._current_span.attributes["cost_usd"] = 0.0052
trace._current_span.attributes["finish_reason"] = "stop"
trace.end_span()

# End top-level span
trace.end_span()

print(json.dumps(trace.to_dict(), indent=2))
```

This pattern is the foundation of tools like LangSmith, Langfuse, and Braintrust. The key insight is that traces are **hierarchical** — spans nest inside spans — mirroring the nested call structure of AI pipelines.

### 2.3 Metrics for AI: Beyond Latency and Throughput

Traditional application metrics focus on the RED method: Rate, Errors, Duration. For AI systems, we need a significantly expanded metrics vocabulary.

**Operational Metrics** (the table stakes):
- **Latency**: p50, p95, p99 for LLM calls, end-to-end request processing, and individual pipeline stages.
- **Error rate**: HTTP errors, timeout rate, rate limit hits, malformed output rate.
- **Throughput**: Requests per second, concurrent LLM calls.

**Token Metrics** (unique to LLMs):
- **Token usage**: Input tokens per request (p50/p95/p99), output tokens per request, total tokens per time window.
- **Context utilization**: Ratio of tokens used to context window size. If you are consistently using 95% of the window, you are one retrieval result away from truncation.
- **Prompt efficiency**: Ratio of "useful" tokens (user query + retrieved context) to "overhead" tokens (system prompt, few-shot examples, formatting). A prompt where 80% of tokens are static boilerplate is ripe for optimization.

**Cost Metrics** (financial observability):
- **Cost per request**: Total API cost for all LLM calls in a single user request. This is the metric that determines unit economics.
- **Cost per user**: Aggregate daily/weekly cost attributed to individual users. Essential for detecting abuse and pricing decisions.
- **Cost per feature**: If your product has a "summarize" feature and a "chat" feature, you need to know which one is burning money.
- **Budget burn rate**: Current spending velocity versus allocated budget, with projections.

The cost per request for a simple single-call system is straightforward:

$$C_{\text{request}} = \frac{n_{\text{input}} \cdot p_{\text{input}} + n_{\text{output}} \cdot p_{\text{output}}}{1000}$$

where $n_{\text{input}}$ and $n_{\text{output}}$ are token counts, and $p_{\text{input}}$ and $p_{\text{output}}$ are price-per-thousand-tokens for the model. For a multi-step agent, the total cost is the sum across all steps:

$$C_{\text{total}} = \sum_{i=1}^{S} C_{\text{request}}^{(i)}$$

where $S$ is the number of steps. Agent cost is inherently variable and often heavy-tailed — most requests cost a few cents, but occasionally an agent enters a loop and burns dollars.

**Quality Metrics** (the hard part):
- **Faithfulness**: Does the response use only information from the provided context? Measured offline via RAGAS-style metrics (Es et al., 2023).
- **Relevance**: Does the response actually answer the user's question?
- **Groundedness**: Can every claim in the response be attributed to a source?
- **User satisfaction**: Thumbs up/down, explicit feedback, implicit signals (did the user ask a follow-up rephrasing the same question?).

```python
import math

# Cost calculation for a multi-model pipeline
PRICING = {
    "gpt-4-turbo": {"input": 10.0, "output": 30.0},    # per million tokens
    "gpt-4o-mini": {"input": 0.15, "output": 0.60},
    "text-embedding-3-small": {"input": 0.02, "output": 0.0},
    "claude-3.5-sonnet": {"input": 3.0, "output": 15.0},
}

def calculate_cost(model: str, input_tokens: int, output_tokens: int) -> float:
    """Calculate cost in USD for a single LLM call."""
    prices = PRICING[model]
    cost = (input_tokens * prices["input"] + output_tokens * prices["output"]) / 1_000_000
    return cost

# Example: a RAG pipeline with embedding + retrieval + generation
embedding_cost = calculate_cost("text-embedding-3-small", input_tokens=50, output_tokens=0)
generation_cost = calculate_cost("gpt-4-turbo", input_tokens=2000, output_tokens=500)
total_cost = embedding_cost + generation_cost

print(f"Embedding cost:   ${embedding_cost:.6f}")
print(f"Generation cost:  ${generation_cost:.6f}")
print(f"Total per request: ${total_cost:.6f}")
print(f"Daily cost (10k requests): ${total_cost * 10_000:.2f}")

# Agent cost distribution simulation
import numpy as np
np.random.seed(42)

# Simulate agent steps: most are 2-4, but some spiral to 10+
steps_per_request = np.random.choice(
    [2, 3, 4, 5, 6, 7, 8, 10, 15],
    p=[0.2, 0.3, 0.2, 0.1, 0.08, 0.05, 0.03, 0.03, 0.01],
    size=1000
)

cost_per_step = 0.012  # average cost per agent step
costs = steps_per_request * cost_per_step

print(f"\nAgent cost distribution (1000 requests):")
print(f"  p50: ${np.percentile(costs, 50):.3f}")
print(f"  p95: ${np.percentile(costs, 95):.3f}")
print(f"  p99: ${np.percentile(costs, 99):.3f}")
print(f"  max: ${np.max(costs):.3f}")
print(f"  total: ${np.sum(costs):.2f}")
```

### 2.4 Logs for AI: The Prompt-Response Record

Traditional logs capture discrete events: "User authenticated", "Database query executed in 45ms", "Payment processed". AI logs must capture something far richer: the full prompt-response pair, including system prompts, few-shot examples, retrieved context, tool call arguments, and tool call results.

This creates a tension. Full prompt-response logging is essential for debugging but creates challenges:

1. **Volume**: A single LLM call can generate 10KB of log data (prompt + response). At 100 requests per second, that is 1MB/s of log data just for LLM interactions — 86GB per day.
2. **PII**: Prompts contain user inputs, which may include personal information, health data, financial details. You must sanitize or encrypt logs.
3. **Cost of storage**: Storing every prompt-response pair indefinitely is expensive. You need retention policies.

The practical approach is **tiered logging**:

- **Tier 1 (always)**: Metadata — model, tokens, latency, cost, finish reason, request ID, user ID.
- **Tier 2 (sampled)**: Full prompt-response pairs for a configurable sample (e.g., 10% of traffic).
- **Tier 3 (triggered)**: Full logging for any request that triggers an anomaly — unusual token count, low quality score, user complaint, guardrail trigger.

This way, you have metadata for every request (cheap to store, fast to query) and full content for the requests that matter.

## 3. OpenTelemetry for AI Systems

OpenTelemetry (OTel) is the CNCF standard for observability instrumentation. It provides a vendor-neutral API for creating traces, metrics, and logs that can be exported to any backend (Jaeger, Datadog, Honeycomb, Grafana). Adapting OTel for AI systems requires defining **semantic conventions** for LLM-specific spans.

The OpenTelemetry community has been developing LLM semantic conventions (still evolving as of 2025). The key attributes for an LLM span are:

| Attribute | Type | Description |
|-----------|------|-------------|
| `gen_ai.system` | string | The AI provider (openai, anthropic, etc.) |
| `gen_ai.request.model` | string | Model name requested |
| `gen_ai.response.model` | string | Model name that actually served the request |
| `gen_ai.request.temperature` | float | Sampling temperature |
| `gen_ai.request.max_tokens` | int | Max output tokens requested |
| `gen_ai.usage.input_tokens` | int | Tokens in the prompt |
| `gen_ai.usage.output_tokens` | int | Tokens in the completion |
| `gen_ai.response.finish_reasons` | string[] | Why generation stopped |

Here is how you would instrument an LLM call using OpenTelemetry concepts:

```python
import time
import json
from dataclasses import dataclass, field
from typing import Dict, List, Any, Optional

@dataclass
class OTelSpan:
    """Simplified OpenTelemetry-style span for AI instrumentation."""
    name: str
    kind: str  # "client", "internal", "server"
    attributes: Dict[str, Any] = field(default_factory=dict)
    events: List[Dict[str, Any]] = field(default_factory=list)
    start_time_ns: int = 0
    end_time_ns: int = 0
    status: str = "unset"  # "unset", "ok", "error"

    def set_attribute(self, key: str, value: Any):
        self.attributes[key] = value

    def add_event(self, name: str, attributes: Dict[str, Any] = None):
        self.events.append({
            "name": name,
            "timestamp_ns": time.time_ns(),
            "attributes": attributes or {}
        })

    @property
    def duration_ms(self) -> float:
        return (self.end_time_ns - self.start_time_ns) / 1_000_000


def trace_llm_call(model: str, messages: list, temperature: float = 0.0) -> OTelSpan:
    """Create an OTel-instrumented LLM call span."""
    span = OTelSpan(name="llm.chat.completion", kind="client")
    span.start_time_ns = time.time_ns()

    # Set Gen AI semantic convention attributes
    span.set_attribute("gen_ai.system", "openai")
    span.set_attribute("gen_ai.request.model", model)
    span.set_attribute("gen_ai.request.temperature", temperature)

    # Estimate input tokens (rough: 4 chars per token)
    prompt_text = json.dumps(messages)
    estimated_input_tokens = len(prompt_text) // 4
    span.set_attribute("gen_ai.usage.input_tokens", estimated_input_tokens)

    # Add prompt event (for sampled logging)
    span.add_event("gen_ai.prompt", {"content_length": len(prompt_text)})

    # Simulate API call
    time.sleep(0.02)
    simulated_output = "This is a simulated response from the model."
    estimated_output_tokens = len(simulated_output) // 4

    # Record response attributes
    span.set_attribute("gen_ai.response.model", model)
    span.set_attribute("gen_ai.usage.output_tokens", estimated_output_tokens)
    span.set_attribute("gen_ai.response.finish_reasons", ["stop"])

    # Calculate cost
    pricing = {"gpt-4-turbo": (10.0, 30.0), "gpt-4o-mini": (0.15, 0.60)}
    if model in pricing:
        input_price, output_price = pricing[model]
        cost = (estimated_input_tokens * input_price + estimated_output_tokens * output_price) / 1_000_000
        span.set_attribute("gen_ai.cost_usd", round(cost, 6))

    span.add_event("gen_ai.completion", {"content_length": len(simulated_output)})
    span.end_time_ns = time.time_ns()
    span.status = "ok"

    return span


# Demo
span = trace_llm_call(
    model="gpt-4-turbo",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Explain observability for AI systems."}
    ],
    temperature=0.0,
)

print(f"Span: {span.name}")
print(f"Duration: {span.duration_ms:.2f}ms")
print(f"Attributes:")
for k, v in span.attributes.items():
    print(f"  {k}: {v}")
print(f"Events: {len(span.events)}")
for event in span.events:
    print(f"  - {event['name']}: {event['attributes']}")
```

## 4. Cost Tracking and Budget Management

Cost is the metric that makes executives pay attention. An unmonitored AI system can quietly spend thousands of dollars in hours. I have personally seen a staging environment left running agent loops overnight that burned $2,400 by morning.

### 4.1 Real-Time Cost Dashboards

A cost tracking system needs three components:

1. **Per-call cost calculation**: Attach a cost to every LLM API call based on model and token counts.
2. **Aggregation**: Roll up costs by user, feature, time window, model, and environment.
3. **Alerting**: Trigger alerts when spending exceeds thresholds.

The cost model for modern LLM APIs is:

$$C = \frac{T_{\text{in}} \cdot P_{\text{in}} + T_{\text{out}} \cdot P_{\text{out}}}{10^6}$$

where $T_{\text{in}}$ and $T_{\text{out}}$ are input and output token counts, and $P_{\text{in}}$ and $P_{\text{out}}$ are prices per million tokens. For cached tokens (increasingly common with prompt caching in Claude and GPT), the input price is discounted:

$$C = \frac{T_{\text{cached}} \cdot P_{\text{cached}} + T_{\text{uncached}} \cdot P_{\text{in}} + T_{\text{out}} \cdot P_{\text{out}}}{10^6}$$

where $P_{\text{cached}}$ is typically $0.1 \times P_{\text{in}}$.

```python
from dataclasses import dataclass, field
from collections import defaultdict
import json

@dataclass
class CostEvent:
    model: str
    input_tokens: int
    output_tokens: int
    cached_tokens: int = 0
    user_id: str = ""
    feature: str = ""

    @property
    def cost_usd(self) -> float:
        pricing = {
            "gpt-4-turbo":       {"input": 10.0,  "output": 30.0, "cached": 1.0},
            "gpt-4o":            {"input": 2.50,   "output": 10.0, "cached": 0.25},
            "gpt-4o-mini":       {"input": 0.15,   "output": 0.60, "cached": 0.015},
            "claude-3.5-sonnet": {"input": 3.0,    "output": 15.0, "cached": 0.3},
            "claude-3-haiku":    {"input": 0.25,   "output": 1.25, "cached": 0.025},
        }
        p = pricing.get(self.model, {"input": 5.0, "output": 15.0, "cached": 0.5})
        uncached = self.input_tokens - self.cached_tokens
        return (uncached * p["input"] + self.cached_tokens * p["cached"] + self.output_tokens * p["output"]) / 1_000_000


class CostTracker:
    def __init__(self, daily_budget_usd: float = 100.0):
        self.events: list = []
        self.daily_budget_usd = daily_budget_usd

    def record(self, event: CostEvent):
        self.events.append(event)

    @property
    def total_cost(self) -> float:
        return sum(e.cost_usd for e in self.events)

    def cost_by_model(self) -> dict:
        by_model = defaultdict(float)
        for e in self.events:
            by_model[e.model] += e.cost_usd
        return dict(by_model)

    def cost_by_user(self) -> dict:
        by_user = defaultdict(float)
        for e in self.events:
            by_user[e.user_id] += e.cost_usd
        return dict(by_user)

    def cost_by_feature(self) -> dict:
        by_feature = defaultdict(float)
        for e in self.events:
            by_feature[e.feature] += e.cost_usd
        return dict(by_feature)

    def budget_alert(self) -> dict:
        pct = (self.total_cost / self.daily_budget_usd) * 100
        status = "ok" if pct < 80 else "warning" if pct < 100 else "critical"
        return {"spent": round(self.total_cost, 4), "budget": self.daily_budget_usd,
                "percent": round(pct, 1), "status": status}


# Simulate a day of traffic
tracker = CostTracker(daily_budget_usd=50.0)

import numpy as np
np.random.seed(42)

models = ["gpt-4o-mini", "gpt-4o", "claude-3.5-sonnet"]
features = ["chat", "summarize", "search"]
users = [f"user-{i}" for i in range(20)]

for _ in range(500):
    model = np.random.choice(models, p=[0.6, 0.25, 0.15])
    tracker.record(CostEvent(
        model=model,
        input_tokens=int(np.random.lognormal(7, 0.5)),  # ~1000 tokens typical
        output_tokens=int(np.random.lognormal(5.5, 0.8)),  # ~250 tokens typical
        cached_tokens=int(np.random.lognormal(6, 1.0)) if np.random.random() > 0.5 else 0,
        user_id=np.random.choice(users),
        feature=np.random.choice(features, p=[0.5, 0.3, 0.2]),
    ))

print("=== Cost Dashboard ===")
print(f"Total: ${tracker.total_cost:.2f}")
print(f"\nBy model:")
for model, cost in sorted(tracker.cost_by_model().items(), key=lambda x: -x[1]):
    print(f"  {model}: ${cost:.4f}")
print(f"\nTop 5 users by cost:")
for user, cost in sorted(tracker.cost_by_user().items(), key=lambda x: -x[1])[:5]:
    print(f"  {user}: ${cost:.4f}")
print(f"\nBy feature:")
for feature, cost in sorted(tracker.cost_by_feature().items(), key=lambda x: -x[1]):
    print(f"  {feature}: ${cost:.4f}")
print(f"\nBudget: {json.dumps(tracker.budget_alert())}")
```

### 4.2 Budget Alerts and Circuit Breakers

Beyond passive monitoring, production systems need **active cost controls**:

- **Soft limits**: Alert when a user exceeds their daily allocation. Log and notify but do not block.
- **Hard limits**: Reject requests when a budget ceiling is reached. Return a 429 or graceful degradation.
- **Circuit breakers**: If cost per request exceeds a threshold (e.g., an agent stuck in a loop), terminate the request and alert.

The circuit breaker pattern for LLM agents is critical. Without it, a single runaway agent can consume your entire daily budget. The check is simple: after each agent step, compare accumulated cost against a per-request ceiling. If exceeded, force termination.

## 5. Dashboard Design for AI Applications

A well-designed AI observability dashboard answers four questions at a glance:

1. **Is the system healthy?** (Latency, error rate, throughput)
2. **Is the system correct?** (Quality scores, hallucination rate, guardrail trigger rate)
3. **Is the system affordable?** (Cost per request, daily spend, budget utilization)
4. **Is the system improving?** (Quality trends, cost trends, user satisfaction trends)

### 5.1 The Critical Metrics Panel

Every AI dashboard should have these metrics front and center:

**Latency (time series, last 24h)**:
- p50, p95, p99 for end-to-end and per-stage (retrieval, generation, post-processing)
- A sudden increase in p99 with stable p50 usually means a subset of requests are hitting a slow path — perhaps a specific model endpoint is degraded or certain queries trigger excessive retrieval.

**Token Usage Distribution**:
- Histogram of input tokens per request, output tokens per request
- Useful for detecting prompt bloat (median input tokens creeping up over time) or model verbosity (output tokens increasing after a prompt change)

**Quality Score Distribution**:
- Histogram of quality scores (if you have a quality evaluator running on sampled traffic)
- Track the percentage of responses below acceptable threshold
- A bimodal distribution (peaks at "good" and "bad") suggests a systematic failure on a subclass of queries

**Cost Burn Rate**:
- Cumulative cost over the day versus budget line
- Projected daily cost based on current rate
- This is the chart that will save you from a $5,000 surprise bill

### 5.2 Anomaly Detection for AI Metrics

Traditional anomaly detection (z-scores, moving averages) works for operational metrics but struggles with AI-specific patterns. Quality scores, for example, can degrade gradually as the underlying data distribution shifts (concept drift) or suddenly when a model provider deploys an update.

For drift detection on continuous metrics, the Page-Hinkley test is practical:

$$m_T = \sum_{t=1}^{T} (x_t - \bar{x}_T - \delta)$$

where $\bar{x}_T$ is the running mean and $\delta$ is a tolerance parameter. An alarm is triggered when $m_T - \min_{t \leq T} m_t > \lambda$ for some threshold $\lambda$. This detects gradual upward shifts — exactly the pattern of slowly degrading quality.

```python
import numpy as np

class PageHinkleyDetector:
    """Detect gradual shifts in a metric (e.g., quality score degradation)."""
    def __init__(self, delta: float = 0.005, threshold: float = 50.0):
        self.delta = delta
        self.threshold = threshold
        self.sum = 0.0
        self.min_sum = float("inf")
        self.count = 0
        self.running_mean = 0.0

    def update(self, value: float) -> bool:
        """Add a new observation. Returns True if drift is detected."""
        self.count += 1
        self.running_mean += (value - self.running_mean) / self.count
        self.sum += value - self.running_mean - self.delta
        self.min_sum = min(self.min_sum, self.sum)
        return (self.sum - self.min_sum) > self.threshold


# Simulate quality scores: stable, then gradual degradation
np.random.seed(42)
stable_scores = np.random.normal(0.85, 0.05, 200)  # normal operation
degraded_scores = np.random.normal(0.85, 0.05, 200) - np.linspace(0, 0.15, 200)  # gradual degradation
all_scores = np.concatenate([stable_scores, degraded_scores])

detector = PageHinkleyDetector(delta=0.005, threshold=20.0)
drift_detected_at = None

for i, score in enumerate(all_scores):
    if detector.update(score):
        drift_detected_at = i
        break

print(f"Degradation started at index: 200")
print(f"Drift detected at index: {drift_detected_at}")
print(f"Detection delay: {drift_detected_at - 200} observations")
```

## 6. Putting It All Together: An Observability Strategy

Here is a practical observability strategy for a production AI system:

**Day 1 (non-negotiable)**:
- Log metadata for every LLM call: model, tokens, latency, cost, request ID
- Track total cost per day with budget alerts
- Monitor latency p50/p95/p99 and error rates

**Week 1**:
- Implement tiered prompt-response logging (metadata always, full content sampled)
- Add trace propagation through your pipeline
- Build a cost dashboard by model, user, and feature

**Month 1**:
- Implement quality scoring on sampled traffic
- Add drift detection on key metrics
- Build a trace explorer UI (or adopt Langfuse/LangSmith)
- Set up circuit breakers for agent loops

**Ongoing**:
- Curate a dataset of "interesting" traces — failures, edge cases, surprising behaviors
- Use this dataset to build regression tests (next lecture: evaluation)
- Monitor evaluation metrics in CI/CD

The companies that do well with AI in production are not the ones with the fanciest models. They are the ones that know, at any moment, exactly what their system is doing, how well it is doing it, and how much it costs. Observability is not a nice-to-have — it is the difference between a demo and a product.

## Summary

AI systems require a fundamentally different approach to observability than traditional software. The three classical pillars — traces, metrics, and logs — must be extended to handle non-deterministic behavior, semantic failures, cascading opacity in agent chains, and cost as a first-class concern.

Traces for AI follow the reasoning process through embedding, retrieval, prompt assembly, LLM invocation, and output validation. Metrics must include token usage, cost tracking, and quality scores alongside latency and throughput. Logs must capture prompt-response pairs with tiered sampling to manage volume and privacy.

Cost tracking is not optional. Without per-call cost calculation, aggregation by user/feature/model, and budget-aware circuit breakers, a single runaway agent can burn your quarterly budget in a weekend. Drift detection on quality metrics catches the slow degradation that human operators miss until customers start complaining.
